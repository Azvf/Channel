# 文档开发与维护规范 v1.0

**核心原则：单一真理源 (SSOT) · 引用即链接 · 自动化守护**

## 1. 黄金法则 (The Golden Rules)

1.  **严禁硬编码 (No Hardcoding)**: 文档中**绝对禁止**出现代码中已定义的具体数值（如 `16px`, `#007AFF`, `300ms`）。

    * ❌ 错误: `圆角: 8px`

    * ✅ 正确: `圆角: var(--radius-md)`

2.  **原则 > 实现 (Principles over Implementation)**: 文档只记录"**为什么**这么设计"和"**心理模型**"。具体的"**怎么实现**"由代码和 Storybook 负责。

3.  **定期消亡 (Expire by Default)**: 每次功能迭代（Feature Kickoff），必须检查关联文档是否过时。过时的文档直接删除，不要标记"待更新"。

## 2. 文档分层架构

| 层级 | 关注点 | 示例 | 维护策略 |
| :--- | :--- | :--- | :--- |
| **L1 稳定层** (Principles) | 架构决策、物理定律、核心交互模式 | `交互手册`, `框架规范` | 极少变动，需架构师通过 |
| **L2 引用层** (Reference) | API 列表、Token 语义、数据库 Schema | `Design Spec`, `API Docs` | **全自动生成**或仅保留指向代码的链接 |
| **L3 指南层** (Guides) | 新人上手、环境配置、故障排查 | `Onboarding`, `Troubleshooting` | 随工具链更新而迭代 |

## 3. 格式规范

* **路径引用**: 必须使用相对路径链接到真实代码文件。

    * 示例: `详见 [SyncService](../src/services/syncService.ts)`

* **图表**: 必须使用 Mermaid 绘制流程图，严禁上传无法编辑的 png/jpg 架构图。

* **架构守护**: 在文档头部标注该文档由哪个工具守护（如 ESLint, Dependency-Cruiser）。

## 4. 废弃与合并

* 如果是针对某个临时问题的 Workaround，请记录在 GitHub Issue 或 Discussion 中，**不要**写进永久文档库。

* 如果两个文档内容重叠度超过 20%，必须合并。
